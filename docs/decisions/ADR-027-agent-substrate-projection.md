## Context

A pi orchestrator running *inside* a rip-cage cage has zero methodology skills today: the cage bakes only `dcg-gate.ts` and bind-mounts only `~/.pi/agent/auth.json`. Claude Code's skills/commands/agents ARE projected (ro-mount of `~/.claude/{skills,commands,agents}` → `.rc-context/*` → `init-rip-cage.sh` symlink). This asymmetry blocks the dotpi self-driving factory: the drover spawns `pi --prompt "/skill:send-it <bead>"` (the ADR-003 D7 keystone), and the spawned sessions dispatch implementer/reviewer sub-agents.

Verified facts (2026-06-16) that shaped this ADR:
1. **Skill discovery is native** — pi scans the filesystem (`readdirSync` of `<PI_CODING_AGENT_DIR>/skills`; any dir with a `SKILL.md` + `description`), invoked `/skill:<name>`. No MCP shim (unlike CC, whose `meta-skill` shim exists because CC ignores the filesystem).
2. **Sub-agent spawning is not native** — pi ships 6 tools (read/bash/edit/find/grep/write) and no spawn primitive; dotpi's `subagent` extension provides it (registers a `subagent` tool wrapping `pi -p` + role loading). So instruction-content alone lets skills *resolve* but not *run* — the `subagent` extension must be projected too.
3. **The `subagent` extension is tool-only** — it calls `pi.registerTool` and registers no `tool_call` handler (`dotpi/agent/extensions/subagent/index.ts:510`), so it cannot interfere with the DCG guard.
4. **pi's `tool_call` blocking is monotonic** — a handler can only return `{block:true}`; there is no "allow" return that overrides a prior block (pi docs/extensions.md). So a co-loaded extension cannot "allow past" the guard. (A separate vector — a malicious handler ordered *after* the guard mutating `event.input.command` post-approval — exists in general but not for the vetted `subagent` extension; it is the epic's validator's job, and relates to rip-cage-olen below.)
5. **Host symlink shape** — `~/.pi/agent/{prompts,roles}` are *relative* symlinks into the dotpi repo; rip-cage's symlink-follow machinery only acts on absolute links and defaults to read-write, so projection must resolve realpaths and force `:ro` explicitly.

This surfaced a question the corpus does not own: **what host substrate is admissible into a cage, and how is it projected?** ADR-019 is pi-specific (config); ADR-005 owns archetypes; ADR-021 D6 / ADR-026 D5 are the seam pattern. None owns the cross-agent admission rule. This ADR does.

## Decisions

### D1 (FIRM) — Agent-substrate admission rests on containment + read-only host-ownership, not on restricting executable code

The cage exists **precisely to contain a coding agent executing arbitrary code** — DCG, the egress firewall, non-root, and the fs sandbox are the safety model. Skills already carry executable scripts (CC and pi alike), so "inert content" was never the real category. **Host agent-substrate — skills (incl. scripts), prompts, roles, AGENTS.md/SYSTEM.md, AND extensions — is admissible to run inside the contained cage**, under two invariants:

- **(a) per-asset `ro`/`rw`, `ro` DEFAULT, host-owned (evolved 2026-06-17 — wlwc D4).** Each substrate asset (per skill subfolder, per extension, per hook) carries a **mount mode**. `ro` (the published default) = locked: the mount is read-only (no cage→host write-back; a compromised cage can't rewrite host substrate) and the source is host-owned / not-agent-writable (closes the ADR-005 D7 pre-stage threat). `rw` (opt-in, power-user, host-configured) = a **live write-through bind-mount** so the agent improves its own skills/extensions in-cage — edits land on the host, versioned via the host's normal git, no separate in-cage git step. This is the property that already justifies mounting `CLAUDE.md`/CC-skills read-only — not "inertness" (`SYSTEM.md`/`AGENTS.md` auto-inject; skills run scripts). The **same `ro`/`rw` knob is what implements the floor-lock** — and as of 2026-06-23 **this decision (D1) owns the floor-lock mechanism** (the former D3 "declared per-agent slot" is dissolved; see D3). The guard wiring is simply an `ro` / root-owned asset on its **own separate load path**, *separate from* any `rw` self-improvement substrate, so an `rw` skill and an `ro` guard coexist in one cage. **Un-tamperability (the agent cannot unlink/replace the wiring) follows from the `ro`/root-owned mount; un-bypassability (the guard loads on every launch) is per-agent and best-effort above containment** — CC's managed-settings auto-loads (airtight); pi's guard rides skippable launch flags and `/usr/bin/pi` stays agent-executable, so its direct-binary bypass is an **accepted residual, deliberately NOT closed via binary-access-control** because that would resurrect olen's per-agent root-own + sudo machinery (containment still bounds blast radius regardless). A mount declared `root_owned_required` is asserted root-owned/non-agent-writable at build (presence-only is fail-open against unlink+recreate).

  - **Per-config security posture (wlwc D7).** Uncrossable-from-inside holds **in the `ro`-default config**: no agent-added/edited substrate exists, so command-gating is genuinely uncrossable from inside for bundled agents (CC: managed-settings hook un-suppressible + deny-wins; pi: guard unwritable + no competing extension addable, so the post-approval-mutation "vector b" stays closed). In **opt-in `rw` mode** the guard wiring is *still* unwritable (separate path) so the guard can't be removed — but pi's post-approval-mutation vector REOPENS for an agent-*edited* `rw` extension; this is a **knowing relaxation backstopped by containment** (the operator opted in on their own machine; command-gating is an accident-guardrail above containment; containment bounds the blast radius). The **fail-closed projected-extension validator** (ADR-005 D11 / D3) is **retained** for `rw`-*projected* extensions (rejecting, at compose time, host-projected extensions that register post-approval-mutating interceptors or shadow the guard); the residual is the runtime-*edited* `rw` extension a compose-time validator cannot catch — named as accepted, not papered over.

  - **Default-posture revision (2026-07-02, rip-cage-px6v — FIRM evolution, human-confirmed).** The DCG recipe's **default** posture no longer declares `--no-extensions` (the OPEN posture, D4). Consequence: the `ro`-default config **no longer closes pi's vector-b by default** — a prompt-injected pi can write an extension into an auto-discovery path (`~/.pi/agent/extensions/`) that pi then auto-loads, potentially sidestepping the command-guard (ADR-024 D2 on-device-harm axis, *in scope*). This is a **knowingly-accepted residual in the default**, added to the same *best-effort-above-containment* list as the direct-binary bypass — **not** a claim that vector-b is closed. The **LOCKED** posture (`--no-extensions`, or `PI_CODING_AGENT_DIR` at a root-owned dir + `defaultProjectTrust:never`) remains a **documented opt-in** that closes vector-b for operators who want the tighter cage. **Inline counter-argument (FIRM rigor, ADR-013 D3):** *moving vector-b closure from default-on to opt-in means the out-of-box cage no longer closes an in-scope prompt-injection vector, and default-takers (the least sophisticated users) are the least likely to opt into locked — a real cost of flipping a security default.* **Rebuttal:** (i) the vector requires a prompt-injected agent to **deliberately craft AND drop** a guard-bypassing extension — near the *out-of-scope* adversarial-coordination line (ADR-024 D5: the injected agent "does not coordinate across layers to bypass"); (ii) **containment (fs/egress/container) is untouched**, so blast radius stays bounded — command-gating is a best-effort accident-guardrail above containment, and pi's guard already carries named residuals here; (iii) the **naive-user risk is mitigated by the cage-composition skill**, which surfaces the posture tradeoff at cage-setup so the open posture is a conscious, agent-relayed choice, never a silent default (**LANDED, rip-cage-p35a.5**: `.claude/skills/configure-cage/SKILL.md`, commit 1c11ce5 — renamed `construct-cage`→`configure-cage` at commit cdead48 — interviews the operator one question at a time, explicitly covering "the DCG guard (including its open-vs-locked extension posture)", before `rc build` ever runs); (iv) the open default is what unblocks pi's **extension-write autonomy** (a core self-improvement workflow, ADR-027 D1 invariant-a `rw` intent) that `--no-extensions` foreclosed, and it dissolves the DCG+herdr composition conflict (D4). Spike-verified end-to-end (rip-cage-px6v, all checks live-green): the open posture keeps DCG **DENYing** destructive commands via the `-e`-loaded guard, herdr detects pi + semantic status flows, and pi auto-loads an agent-written extension. **Invalidation of the open default:** if injected agents in practice routinely exploit the reopened extension vector to cause on-device harm the guard would have stopped, revisit the default (flip back to locked-default, keeping open as opt-in).

  - **Note:** read-only projection is *strictly safer* than the former status quo, where the baked guard's own directory was agent-writable (rip-cage-olen). That per-agent root-own-of-the-whole-`extensions/`-dir + sudo-dance machinery is **retired** (rip-cage-wlwc.4): the pi guard now lives root-owned on its OWN separate load path `/etc/rip-cage/pi/dcg-gate.ts` (file + parent dir root-owned), loaded via `pi -e` (plus `--no-extensions` under the **LOCKED** opt-in; the **OPEN** default omits it — see D1 "Default-posture revision"), and the agent's `extensions/` dir is agent-owned — a dir cannot be both root-locked and agent-writable, which is exactly why olen's approach conflicted with `rw` extensions.
- **(b) cannot weaken the welded safety floor.** A *projected* extension is admissible only when it cannot disable or bypass the floor. For the interim (D2) this is established **per-extension by vetting** the specific extension projected (e.g. the `subagent` extension is verified tool-only, no `tool_call` interceptor — fact #3) combined with **read-only** projection (the agent cannot mutate the projected copy) and pi's **monotonic block** (fact #4). The *general* mechanism — a fail-closed validator that rejects any projected extension hook that mutates a command post-approval, shadows the guard, or registers a bypassing interceptor — is built by the D3 seam (the MULTIPLEXER hook-bounds checks at `rc:6766+` are the template). This ADR does NOT assume "the floor is already uncrossable from inside": that holds for the config vector (ADR-025) and, as of **rip-cage-olen (retired and re-done as the separate-path floor-lock, rip-cage-wlwc.4)**, for the **pi** agent-writable-extension vector (the baked guard is root-owned on its own separate load path `/etc/rip-cage/pi/`; **pre-flip** — before the 2026-07-02 OPEN-default revision — pi launched `--no-extensions -e`, so the agent could not replace the guard nor get a competing post-approval-mutating extension loaded. Under the current **OPEN** default this vector-b closure is a **LOCKED**-opt-in property, not the default; see D1 "Default-posture revision" for the now knowingly-accepted residual). The **CC** hook-registration vector (rip-cage-r9n4, **closed 2026-06-22** via managed-settings) is now also closed. Making the floor uncrossable-from-inside was a sibling precondition, not a premise of D1; for pi's baked-guard-write vector it is now satisfied (the *projected*-extension case stays per-extension-vetted until the D3 validator).

`CONFIG` (settings/keybindings/trust/themes/models), `CREDENTIALS` (`auth.json` — the lone RW mount, ADR-019 D1), and `RUNTIME-STATE` (sessions/git/npm/logs) stay **cage-owned** — for config-integrity, secret, and mutable-state reasons respectively, unrelated to whether they execute.

This generalizes ADR-019 D1: the host→cage surface is no longer "only auth.json" but "auth.json (RW) + read-only host-owned agent-substrate." The previously-hardcoded CC projection is retroactively the canonical instance.

**Inline counter-argument (FIRM rigor, ADR-013 D3):** *ADR-019 D1 narrowed the pi host→cage surface to only `auth.json` to shrink attack surface under ADR-024; admitting skills/extensions re-expands it, extensions are executable code that could subvert the guard (verification even confirmed the guard was self-disablable — olen), and now (2026-06-17) admitting `rw` substrate opens a cage→host write-back path a compromised cage could use to rewrite host methodology that auto-loads in the next host session.* **Rebuttal:** ADR-024's threat is the agent following hostile instructions from **attacker-controllable channels it fetches** (web/READMEs/MCP/workspace) — not operator-owned, host-controlled methodology substrate. "Executable code in the cage" is the cage's premise, not a new risk. The guard-subversion concern is handled without assuming an uncrossable floor: in the `ro`-default config the projected extension is read-only (the agent can't tamper with it), it is vetted tool-only / validator-bounded (registers no interceptor), and pi's block is monotonic (no allow-past). On the new `rw` surface: it is **opt-in, host-configured, per-asset** — the `ro` published default has *no* write-back path; an operator who enables `rw` on their own machine knowingly trades a backstop for the self-improvement flywheel, the guard wiring stays unwritable regardless of mode (so `rw` never reaches the guard), and the cage→host edit is bounded by containment + the operator's own opt-in (wlwc D4/D7). A future isolated-copy + reviewed-promote flavor (sourcing from the host's git, not a live bind-mount) is the named mitigation if the live flavor proves too sharp — tracked, not built now. The agent-writable-guard weakness (olen) is *pre-existing and independent* of this projection; olen's fix (**done — retired and re-done as the separate-path floor-lock, rip-cage-wlwc.4**: the pi guard is root-owned at `/etc/rip-cage/pi/dcg-gate.ts`, loaded `-e` — plus `--no-extensions` under the **LOCKED** opt-in, the **OPEN** default omits it, see D1 "Default-posture revision") closes the baked-guard-write vector, and the D3 validator closes the *projected*-extension case generally. The real anomaly the narrow D1 reading created is the CC/pi asymmetry.

**What would invalidate D1:** if a *read-only, vetted, tool-only* projected extension could still weaken the floor (e.g. pi's block turns out non-monotonic, or extension load can preempt the guard despite vetting), invariant (b)'s interim basis fails and extensions must be baked-root-owned only, not mountable. OR (the `rw` surface, wlwc D4/D7): if live `rw` write-through proves a real cage→host escalation **even on the operator's own machine** (not merely a theoretical write-back), the live `rw` flavor is replaced by isolated-copy + reviewed-promote; OR if the published default is ever shipped `rw`, or the `ro`-default's vector-b closure does not actually hold (pi loads agent-writable extensions even in `ro` mode), the default-safe posture fails.

| Alternative | Rejected because |
|---|---|
| Keep ADR-019 D1 "only auth.json"; bake everything | `reasoned:` methodology substrate changes constantly (edited this session) → image rebuild per edit; and leaves the CC/pi asymmetry unprincipled. |
| Exclude executable extensions; mount only "inert" content | `direct:` skills already carry executable scripts (CC + pi), and pi can't spawn sub-agents without the `subagent` extension (verified fact #2) — so "inert-only" neither matches reality nor unblocks the factory. |
| All-`ro` substrate, never writable (the former invariant (a)) | `direct:` (reframed 2026-06-17 — wlwc D4) blocks the self-improvement flywheel: an agent improving its own skills/extensions in-cage is a core maintainer workflow and a product feature that read-only-from-host forecloses. `rw` is now **admitted, opt-in, per-asset, `ro`-default** — the all-`ro` rule is too coarse to express "rw skills + ro guard in one cage," which is exactly the floor-lock requirement. |
| Admit substrate read-write *globally / by default* (blanket parity with auth.json) | `reasoned:` a blanket-`rw` default would open an unbounded cage→host write-back path for every cage. The admitted form is **per-asset, host-configured, `ro` by default** — write-through is a knowing operator opt-in on their own machine, backstopped by containment, with the guard wiring unwritable regardless of mode (wlwc D4/D7); a global default-`rw` is what stays rejected, not `rw` itself. |

### D2 (FLEXIBLE — being RETIRED by the now-FIRM D3 seam, 2026-06-17) — pi substrate (instruction-content + the `subagent` extension) projected as a hardcoded per-bundled-agent block, symmetric with CC (interim)

> **Retirement status (2026-06-17, wlwc):** D3 has been lifted EXPLORATORY→FIRM, so the seam that absorbs this interim is now being **built**, not deferred. D2 is **acknowledged interim debt on a now-active retirement path** — the hardcoded CC + pi projection blocks migrate onto the D3 generic `TOOL` + `ro`/`rw`-mount composition (no agent archetype — mechanism reversed 2026-06-23), and this decision retires when that migration lands (no half-migrated state — pi's olen root-own-dir machinery is re-done onto the unwritable-separate-path model in the same program; wlwc D5/D6). The shipped interim below remains an accurate description of current state until the migration completes.

Project pi's instruction-content (skills, prompts, roles, AGENTS.md, SYSTEM.md/APPEND_SYSTEM.md seam) **and** dotpi's `subagent` extension into the cage, read-only, so `/skill:send-it` and `/drover` both *resolve and run*. The shipped interim (rip-cage-kstk) implements this as a **hardcoded per-bundled-agent projection block in `rc`** — a presence-gated (`[[ -d ~/.pi/agent ]]`) loop that is data-driven over *that agent's asset table* (`{asset → cage-path}` tuples), with **no `if/elif` branch on agent identity** beyond the presence gate. This is **symmetric with the pre-existing hardcoded CC projection block** (the `~/.claude/{skills,commands,agents}` mounts), which is the same shape.

This is **D12-clean by classification, not by manifest-derivation**: ADR-005 D12 forbids `rc` naming a specific *optional* tool (the herdr-leak anti-pattern); `claude` and `pi` are **bundled cage-infra** (per D1, agent-substrate projection is cage-infrastructure), so a hardcoded bundled-agent block does not trip D12. The interim is therefore **NOT yet manifest-derived** — `rc` does not read the seeded manifest to discover which agents declare substrate; that is the **D3 target** (agents compose from generic `TOOL` + `ro`/`rw` mounts; no agent archetype — mechanism reversed 2026-06-23).

Mechanism per asset: resolve host realpath (the dotpi symlinks) → bind-mount `:ro` → `/home/agent/.rc-context/pi-<asset>` → ADR-023 denylist check (warn-and-skip, incidental tier) → symlink into `~/.pi/agent/<asset>` (pi discovers via `PI_CODING_AGENT_DIR`). The `subagent` extension is projected **selectively** (just that extension, vetted tool-only per fact #3) so the baked `dcg-gate` is not shadowed.

**Rationale:** projecting a bundled agent's substrate is cage-infrastructure, not optional-tool composition; the **bundled-vs-optional classification** (not manifest-derivation) is what keeps the hardcoded block D12-clean. This is **acknowledged interim debt** — pre-seam hardcode, NOT "floor", NOT a D12 exception — to be absorbed by D3. (A fresh-context reviewer read the hardcoded `pi` block as a D12 violation precisely because the earlier D2 prose promised manifest-derivation the interim never implemented; this revision removes that trap.)

**What would invalidate D2:** D3's seam ships → CC + pi migrate onto it and this retires.

### D3 (FIRM — lifted 2026-06-17; mechanism reversed 2026-06-23, wlwc) — agents, substrate, and guards compose from the generic `TOOL` + per-asset `ro`/`rw` mount seams; no agent archetype

> **Mechanism reversal (2026-06-23, wlwc — human-confirmed).** The *goal* of D3 is unchanged: agents are composable, zero-`rc`-edit, published for heterogeneous harnesses. The *mechanism* is reversed. The earlier D3 introduced a dedicated **CONTENT-MOUNT agent archetype** + an `agents.enabled` agent-selection key + a manifest-derived agent allowed-set + a baked per-agent registry/label + a **declared per-agent floor-lock slot**. That is all retired: an agent is **not** a first-class `rc` concept — it is a **composition of two generic seams** (a `TOOL` entry + per-asset `ro`/`rw` mounts), and a guard is the same shape. No `AGENT` archetype, no `agents.enabled`, no per-agent floor-lock slot. (The bespoke archetype was partially shipped then reverted — rip-cage-wlwc.2.1.)

**An agent = `TOOL` + `ro`/`rw` mounts.** The binary and its launch wrapper are a `TOOL` entry; its skills/prompts/roles/extensions/config project as per-asset `ro`/`rw` mounts (D1's seam — `ro` default, host-owned; `rw` opt-in write-through). Selection is by composing the agent's fragment into the config at build time; there is **no per-cage runtime agent selection** (a deliberate simplification — re-addable later as *generic* per-cage fragment selection, never agent-specific). Adding an agent (codex, crush, …) is `TOOL` entries + mount declarations + a config-composition selection, with **zero `rc` edits and zero archetype**.

**A guard = `TOOL` + `ro`-mounted wiring; the floor-lock is the `ro` mount (mechanism owned by D1).** The guard *engine* (e.g. `dcg`, built from source per ADR-005 D11) is an ordinary `TOOL`; the guard *wiring* (the CC managed-settings entry / pi extension) is an `ro` / root-owned mount asset on its **own separate load path**. So `dcg` and `block-ssh-bypass` demote from base-image bakes to **composable command-guard recipes** (ADR-025 D2, ADR-026 D2). Un-tamperability follows from the `ro`/root-owned mount; un-bypassability is per-agent, best-effort above containment (CC airtight via managed-settings; pi leaves a direct-binary residual — see D1, NOT closed via binary-access-control, to keep olen retired). **Two validators live on the mount seam** (not on an archetype): (a) a mount declared `root_owned_required` is asserted root-owned/non-agent-writable (else the `ro` floor-lock is a lie); (b) an `rw`-projected extension must pass a **fail-closed floor-protection check** (ADR-005 D11 — code-inspection reject of guard-shadow / post-approval command mutation / bypassing interceptor), retained for `rw`-projected extensions (D1/wlwc-D7).

**Default distribution is a config choice, not a code/seam blessing.** The published image (brew→GHCR) is built from a default config that composes the CC + pi + `dcg` (+ `ssh-bypass`) recipe fragments, so out-of-box `rc up` works as before — but `rc` code, the seam, and the default *manifest* bless nothing (parallel to the egress observe-mode default). **Composition is performed by an agent reading the recipe, not by an `rc` mechanism** — there is no `compose:`/`include:` directive (ADR-005 D12, agentic-composition premise). olen's per-agent root-own-dir + sudo machinery is **retired** (rip-cage-wlwc.4; a dir cannot be both root-locked and agent-writable, which conflicts with `rw` self-improvement); its job is done by the root-owned floor-lock asset on a separate load path (for pi, the guard is `/etc/rip-cage/pi/dcg-gate.ts`, loaded `-e` by default — plus `--no-extensions` under the **LOCKED** opt-in, see D1 "Default-posture revision"; for a baked asset the realization is "root-owned file + root-owned parent dir on a cage-owned system path" — the baked analog of the `ro`-mount, both giving ownership-not-presence un-tamperability). **A composed default image asserts its floor-guards are *present+loaded*** via the declared safety-stack (each guard recipe declares the generic `required: true` + `assert_loaded` property — the guards are the first consumers of a property any TOOL may carry, not a special-cased category; codegen bakes a root-owned `/etc/rip-cage/safety-stack-asserted` declaration the in-cage safety-stack test runs generically and fails-loud against — **ADR-005 D13**), so a default cage that silently lost a composed guard fails loud while `rc` stays tool-agnostic (rip-cage-wlwc.6 + rip-cage-m8zc).

**Guard-hook home + per-composition assembly (rip-cage-wlwc.12).** For an agent whose guard wiring is a *hook list* (CC's PreToolUse hooks), the canonical secure home is the **root-owned managed-settings file** (`/etc/claude-code/managed-settings.json`): it auto-loads, is un-suppressible, and the agent cannot strip it. Merging the hook into the agent-writable `~/.claude/settings.json` is a **weaker manual fallback** a prompt-injected agent (ADR-024) could remove. The assembled hook *set* is **per-composition DATA, not hardcoded**: the generic `examples/claude` recipe ships the `dcg` hook only, while the default published composition (`dist/default-tools.yaml`) provisions a managed-settings carrying **both** the `dcg` and `ssh-bypass` hooks — assembled by the maintainer/agent authoring that composition, never by an `rc` `compose:` mechanism (ADR-005 D12). This is the hook-list instance of the same floor-lock rule: the guard wiring is a root-owned asset on its own un-tamperable load path, and *which* hooks compose is a config choice.

**Nuance:** agent-substrate is **additive** (all present agents project simultaneously, TOOL-like set semantics), unlike single-select multiplexer/mediator. This is the "enable agent extensions, cleanly" design — extensions are admitted as *validated* providers, not excluded. The validator here is what makes invariant (b) structural rather than per-extension-vetted, and it is **retained for `rw`-projected extensions** (D1/wlwc-D7).

Anchors a follow-on **epic** (compose agents/guards from `TOOL` + `ro`/`rw` mounts → migrate CC → migrate pi off the D2 interim debt → re-do the olen floor-lock as the `ro`-mount on a separate path, D1). Relates to rip-cage-olen (the guard-integrity fix the validated seam assumes — **retired and re-done as the separate-path floor-lock, rip-cage-wlwc.4**: pi guard root-owned at `/etc/rip-cage/pi/`, not inside the agent's `extensions/` dir) and rip-cage-r9n4 (**closed 2026-06-22 for CC** via managed-settings as the first floor-lock instance, wlwc D6).

**Rationale (lift warrant):** rip-cage is **published for heterogeneous harnesses** — the maintainer runs CC + pi, but publishes rip-cage for users on codex, crush, and other agents — so "pluggable agents" is a product requirement, not speculative generality. The rule-of-two objection to building the seam (the EXPLORATORY basis) no longer holds: the third/fourth/fifth agents are the published users' harnesses. And the per-agent floor-lock machinery (olen) does not generalize — it root-owned a whole directory + needed a sudo dance to place even one extension symlink, which does not scale to N projected/editable extensions and directly conflicts with letting agents improve their own extensions in-cage; it was about to be copy-pasted into r9n4 as a second snowflake. The generic `TOOL` + `ro`/`rw`-mount composition (floor-lock = the `ro` mount on a separate path, D1) dissolves both — without any agent archetype.

**Alternatives considered:**

| Alternative | Rejected because |
|---|---|
| Keep CC/pi hardcoded interim, defer the seam (the EXPLORATORY status quo) | `reasoned:` the publish-for-other-harnesses requirement removes the "no third agent" gate — the third agent is the users' harnesses, so the seam is load-bearing now, not speculative. |
| Bake a fixed agent enum into `rc` | `direct:` ADR-005 D12 forbids naming optional tools in source; agents become manifest-declared providers with zero `rc` edits to add one. |
| Keep olen's per-agent root-own-dir + sudo-dance floor-lock | `direct:` does not generalize to N agents/extensions, conflicts with `rw` extensions (a dir cannot be both root-locked and agent-writable), and was about to become a second snowflake at r9n4 — replaced by the `ro`-mount floor-lock on a separate load path (D1), no per-agent machinery. |

**What would invalidate D3:** if the agent-provider contract cannot express a real target harness's wiring (some agent has no command-interception mechanism at all), that agent gets containment-only (no command-gating) — an accepted gap (containment is the floor; the command-guard is a best-effort accident-guardrail per agent); but if that is the *common* case, the seam's command-guard value is reduced.

### D4 (mechanism FLEXIBLE; the no-hardcoded-cross-recipe-path *principle* FIRM as the launch-leg instance of ADR-005 D12 / D3 — 2026-06-29, revised twice: post-review added a root-owned glob dir + admission validator; **re-revised same day post-dogfood (human-confirmed) to the simpler manifest-declared mechanism below** — launch args/extensions are declared in the composed manifest fragments and assembled by `rc build`; the runtime glob dir and the launch-leg admission validator are dropped as over-built) — Per-tool launch composition: `rc build` assembles the launch wrapper from manifest-declared `launch_args` + extension mounts contributed by the composed recipe fragments; no hardcoded cross-recipe paths, no runtime contribution dir

D3 settled that an agent = `TOOL` + `ro`/`rw` mounts and that composition is agent-performed, but at the time this decision was opened the *shipped launch leg* still hardcoded its composition: `examples/pi/pi-wrapper.sh` forced `--no-extensions -e <dcg-gate>` and carried a single hardcoded `SUBAGENT_EXT` slot. That hardcoding was the D2 interim debt in its launch form — a second recipe (e.g. herdr) that needed a pi extension loaded could not contribute without editing the wrapper, and `--no-extensions` silently dropped any extension installed the ordinary way (`herdr integration install pi` → agent-owned `~/.pi/agent/extensions/`, a dir `--no-extensions` tells pi to ignore). D4 specifies the launch-composition mechanism that retires this — **and, per the LANDED stamp above (rip-cage-l72i), it has since shipped**: `examples/pi/pi-wrapper.sh` is now the generic manifest-assembled template (no hardcoded extension path, no `SUBAGENT_EXT` slot; see D4 body below).

**The launch wrapper is assembled by `rc build` from manifest-declared launch arguments and extension mounts** — it hardcodes **no** individual extension paths and names **no** recipe. Each recipe fragment declares its launch contribution *in the manifest, where the tool is already configured*: a generic `launch_args:` list per fragment plus the extension's `ro` mount. The DCG fragment declares `-e <dcg-gate>` (its **OPEN default** posture, 2026-07-02 — the **LOCKED** opt-in variant additionally declares `--no-extensions`; see the `--no-extensions`-are-recipe-contributed note below and D1 "Default-posture revision"); a herdr-pi fragment declares `-e <herdr-ext>` + the herdr socket mount; the `subagent` fragment declares `-e <subagent>`. `rc build` concatenates the `launch_args` across the composed fragments **in fragment order** (the guard fragment composed first ⇒ guard loads first) and bakes a generic wrapper that simply execs the real binary with the assembled args. Adding the Nth extension is a `launch_args` + mount declaration in a legible recipe fragment — zero wrapper edits, zero `rc` source edits. The `launch_args` field is `rc` **owning the composition interface** (ADR-005 D12), the same way it already owns mount declarations — not blessing a tool.

There is **no runtime contribution directory**: the extension set is a *build artifact assembled from a host-authored manifest*, not a dir scanned at launch. This is the load-bearing simplification over the post-review glob mechanism — see the validator paragraph and the Alternatives table below. Extension generation, where a tool ships a generator, still goes through the tool's **public CLI** (`herdr integration install pi`), honoring ADR-006 D8 ("wire tool↔tool through public CLIs, never hand-author another tool's internals"). (*Verified 2026-06-29 (rip-cage-l72i.4):* `herdr integration install pi` (v0.7.0) accepts **no** output-directory flag — it always writes to `<PI_CODING_AGENT_DIR>/extensions/herdr-agent-state.ts`. So the recipe runs the public CLI (file 100% herdr-authored, D8-clean) then **relocates** the generated file post-generate to the root-owned cage path — a **named, bounded D8 deviation** (only the location is cage-dictated; herdr could close it fully by adding `--output-dir`). This is the minimal deviation consistent with D8.) `--no-extensions`, when contributed (by the DCG fragment's **LOCKED** opt-in variant — the OPEN default omits it as of 2026-07-02), keeps the *agent-writable* auto-discovery paths (`/workspace/.pi/extensions/`, `~/.pi/agent/extensions/`) dead; the manifest-declared `-e` set is the vetted load path, so the guard's tamper-proofing (D1) and a contributed herdr extension coexist in one cage. The `examples/` recipes are **inspiration** — concrete fragments an agent reads to learn how to declare a tool's launch composition — not validated machinery and not a runtime-scanned contribution surface.

**No launch-leg code-inspection validator (the post-review addition is dropped as over-built).** The post-review revision added a fail-closed validator that code-inspected every glob-admitted extension for guard-shadow / post-approval mutation / bypassing-interceptor patterns. That validator only earned its keep under the *runtime glob* model, where the wrapper would load whatever files were present in a dir at launch. Under the manifest-declared mechanism it is unnecessary, and the floor is carried without it by: (i) the guard wiring is root-owned on its own load path (D1) — the agent cannot replace it; (ii) **under the LOCKED opt-in posture only**, `--no-extensions` (manifest-declared) keeps the agent-writable auto-discovery paths dead — the **OPEN** default (D1 "Default-posture revision") omits this flag, so under the default posture the floor rests on (i) and (iii) alone and the reopened auto-discovery vector is a named accepted residual there; (iii) the loaded extension set is a **build artifact assembled from a host-authored manifest**, and `rc build` is host-side — so the prompt-injection vector the validator guarded against (a contained agent dropping `evil.ts` into a scanned dir) **cannot reach it** (ADR-024 D4: host-side composition is out of scope). What remains is an *operator* composing a guard-shadowing extension into the manifest alongside DCG — a **composition choice**, the configuring agent's job (ADR-005 D12), bounded by containment, and the **same accepted-residual shape** as the already-accepted "compose no DCG ⇒ no guard." A guard-shadow regression in the *published default* composition is caught by the safety-stack present+loaded assert (ADR-005 D13) and the live integration test — not by static TypeScript inspection, which D4's own invalidation clause concedes can miss it. **The FIRM D1/D3 fail-closed validator for `rw`-*projected* (live write-through, agent-editable) extensions is untouched and still applies** — this evolution removes only the launch-leg glob-admission validator, which the now-removed glob required.

**Named residuals (per D1's honesty discipline):** (a) in the *no-DCG* composition there is no command guard at all, so an auto-discovered/agent-dropped extension runs unguarded — accepted by design (drop the recipe, drop the guard; containment still bounds blast radius). (b) The direct-binary bypass (`/usr/bin/pi` skipping the PATH-shadow wrapper, and with it whatever launch args the wrapper would have assembled — under the **LOCKED** opt-in, `--no-extensions`; under the **OPEN** default, just `-e <dcg-gate>`) is **inherited from D1 unchanged** — an accepted residual deliberately not closed via binary-access-control, to keep olen retired; the PATH-shadow delivers the same best-effort un-bypassability D1 grants, no stronger. (c) An *operator* composing a guard-shadowing extension into the manifest alongside DCG runs it — a composition choice (ADR-005 D12), bounded by containment, the same accepted shape as (a); no build-time static check polices the operator's own manifest in the launch leg (the `rw`-projected-extension validator of D1/D3 is a separate surface and is retained).

**`--no-extensions` (and guard-hardening launch postures generally) are recipe-contributed `launch_args`, not baked invariants — and as of 2026-07-02 (rip-cage-px6v) the DCG recipe's DEFAULT posture is OPEN: it does NOT declare `--no-extensions`.** The **default** DCG fragment declares only `-e <dcg-gate>` — the guard still loads and still DENYs destructive commands (spike-verified) — while a **locked variant** (documented opt-in) *adds* `--no-extensions` for operators who want the agent-writable auto-discovery paths (`/workspace/.pi/extensions/`, `~/.pi/agent/extensions/`) dead (closing vector-b, D1). Rationale for open-default: it lets pi auto-discover and **write its own extensions** (autonomy) and makes an ordinarily-installed herdr integration *just work*; the tradeoff (reopened vector-b) is a knowing residual surfaced to the operator by the construction skill (see D1 "Default-posture revision"). Drop the DCG fragment entirely → also no guard. **The DCG+herdr conflict that motivated D4 dissolves under the open default** — herdr's extension auto-discovers, no `-e` gymnastics required (the `-e` path still works and is how the locked posture loads vetted extensions). The README of each fragment documents its posture. (This does not touch the FIRM no-hardcoded-cross-recipe-path principle — a recipe still contributes its launch args by manifest declaration; only *which* args the default DCG fragment contributes changed.)

**Ordering / conflict across *non-floor* contributions is the configuring agent's job** (ADR-005 D12 agentic-composition premise): `rc build` provides deterministic ordering (fragment order in the composed manifest; the guard fragment composed first ⇒ guard loads first), so the agent resolves ordering only among non-floor contributions by ordering its fragments. rip-cage does **not** pre-enumerate every recipe combination. Whether the assembled wrapper is an `rc`-provided generic shim or a per-tool wrapper is left to the retirement epic — the load-bearing rule (the **FIRM principle**) is "**no hardcoded cross-recipe paths in any launch leg; recipes contribute their launch args + extensions by manifest declaration, assembled at build.**" The PATH-shadow property (every invocation — agent-typed, multiplexer-spawned, scripted — hits the composed launch, which pure config-file launch cannot guarantee) is preserved either way and is itself why a launch *wrapper/shim* exists at all.

**Rationale:** the shipped per-wrapper hardcoding reproduces, one layer down, the exact overfit ADR-005 D12 forbids — a wrapper that names `subagent` (or would name `herdr`) is a blessed-tool slot in disguise. Declaring launch args + extensions **in the manifest** makes the launch leg compose the same way the rest of the agent substrate already does under D1/D3 (mounts are manifest-declared; launch args join them), so adding the Nth extension is a fragment declaration, not a wrapper edit and zero `rc` source edits. The manifest-declared mechanism is preferred over the post-review runtime glob + admission validator because it keeps the *examples as inspiration* (not runtime-scanned machinery), puts launch config where tool config already lives, and closes the agent-drops-`evil.ts` vector by construction (build artifact from a host-authored manifest) rather than by a brittle static-inspection gate above containment.

**What would invalidate D4 (the mechanism — the FIRM no-hardcoded-path principle is not at stake):** if a guard-shadowing extension composed into a *published default* image slips past the safety-stack present+loaded assert (ADR-005 D13) **and** the live integration test and reaches a shipped cage undetected, the dropped launch-leg check returns — scoped to the published default composition, not to arbitrary operator contributions. OR if a future feature re-introduces a **runtime** contribution surface (a dir scanned at launch rather than a build artifact from a host-authored manifest), the "build is host-side, agent can't reach it" argument no longer holds and the admission validator must return for that surface. OR if fragment-order proves insufficient for a real ordering constraint (a contribution that must load before another within the same fragment-order slot), an explicit per-contribution priority field replaces bare fragment-order — a refinement, not a reversal. OR if a real target tool's launch genuinely cannot accept an assembled flag list at all (no `-e`-equivalent, no enumerable extension flag), that agent's command-guard degrades to containment-only — an accepted gap per D3's invalidation clause.

## Consequences

- D2 *actually unblocks* the in-cage drover dogfood (dotpi-3bi.8/.4) — skills resolve AND sub-agent dispatch runs — without weakening the floor.
- The CC/pi asymmetry was named interim debt with a now-**discharged** (D3 FIRM, 2026-06-17; mechanism reversed 2026-06-23) retirement path: compose agents/guards from `TOOL` + `ro`/`rw` mounts → migrate CC → migrate pi off the D2 interim → re-do the olen floor-lock as a root-owned asset on a separate path (D1). All landed (rip-cage-wlwc.2/.3/.4).
- Spawned child `pi` processes inherit the guard (they invoke the root-owned `pi` wrapper, which prepends `-e /etc/rip-cage/pi/dcg-gate.ts` under the **OPEN** default — the **LOCKED** opt-in additionally prepends `--no-extensions`); the agent-writable-guard weakness once tracked in rip-cage-olen is **closed for pi** (root-owned guard on its OWN separate load path `/etc/rip-cage/pi/`, so the guard itself is unreplaceable regardless of posture; the agent's `extensions/` dir is agent-owned and — under the **OPEN** default — auto-scanned (the **LOCKED** opt-in disables that scan), so a competing auto-loaded extension is the reopened vector-b now accepted as a residual under the open default, D1 "Default-posture revision" — rip-cage-wlwc.4).
- The "is executable substrate admissible?" question is resolved (yes, under containment + read-only/host-owned + per-extension vetting now / validator later), dissolving the earlier executable-exclusion.
