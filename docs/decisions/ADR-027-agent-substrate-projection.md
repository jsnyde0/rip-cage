# ADR-027: Agent-Substrate Projection into the Cage

Status: PROPOSED — D1 FIRM (human-confirmed 2026-06-16; evolved 2026-06-17 — per-asset ro/rw, ro default), D2 FLEXIBLE (being RETIRED by the now-FIRM D3 seam), D3 FIRM (lifted EXPLORATORY→FIRM 2026-06-17 — wlwc: published for heterogeneous harnesses, the third agent is the users' harnesses)
Date: 2026-06-16 (revised 2026-06-17 — wlwc: D1 evolved to per-asset ro/rw + per-config security posture; D2 marked retiring; D3 lifted to FIRM. Prior 2026-06-17 — D2 prose revised to describe the shipped hardcoded-symmetric interim accurately)

## Context

A pi orchestrator running *inside* a rip-cage cage has zero methodology skills today: the cage bakes only `dcg-gate.ts` and bind-mounts only `~/.pi/agent/auth.json`. Claude Code's skills/commands/agents ARE projected (ro-mount of `~/.claude/{skills,commands,agents}` → `.rc-context/*` → `init-rip-cage.sh` symlink). This asymmetry blocks the dotpi self-driving factory: the drover spawns `pi --prompt "/skill:send-it <bead>"` (the ADR-003 D7 keystone), and the spawned sessions dispatch implementer/reviewer sub-agents.

Verified facts (2026-06-16) that shaped this ADR:
1. **Skill discovery is native** — pi scans the filesystem (`readdirSync` of `<PI_CODING_AGENT_DIR>/skills`; any dir with a `SKILL.md` + `description`), invoked `/skill:<name>`. No MCP shim (unlike CC, whose `meta-skill` shim exists because CC ignores the filesystem).
2. **Sub-agent spawning is not native** — pi ships 6 tools (read/bash/edit/find/grep/write) and no spawn primitive; dotpi's `subagent` extension provides it (registers a `subagent` tool wrapping `pi -p` + role loading). So instruction-content alone lets skills *resolve* but not *run* — the `subagent` extension must be projected too.
3. **The `subagent` extension is tool-only** — it calls `pi.registerTool` and registers no `tool_call` handler (`dotpi/agent/extensions/subagent/index.ts:510`), so it cannot interfere with the DCG guard.
4. **pi's `tool_call` blocking is monotonic** — a handler can only return `{block:true}`; there is no "allow" return that overrides a prior block (pi docs/extensions.md). So a co-loaded extension cannot "allow past" the guard. (A separate vector — a malicious handler ordered *after* the guard mutating `event.input.command` post-approval — exists in general but not for the vetted `subagent` extension; it is the epic's validator's job, and relates to rip-cage-olen below.)
5. **Host symlink shape** — `~/.pi/agent/{prompts,roles}` are *relative* symlinks into the dotpi repo; rip-cage's symlink-follow machinery only acts on absolute links and defaults to read-write, so projection must resolve realpaths and force `:ro` explicitly.

This surfaced a question the corpus does not own: **what host substrate is admissible into a cage, and how is it projected?** ADR-019 is pi-specific (config); ADR-005 owns archetypes; ADR-021 D6 / ADR-026 D5 are the seam pattern. None owns the cross-agent admission rule. This ADR does.

## Decisions

### D1 (FIRM) — Agent-substrate admission rests on containment + read-only host-ownership, not on restricting executable code

The cage exists **precisely to contain a coding agent executing arbitrary code** — DCG, the egress firewall, non-root, and the fs sandbox are the safety model. Skills already carry executable scripts (CC and pi alike), so "inert content" was never the real category. **Host agent-substrate — skills (incl. scripts), prompts, roles, AGENTS.md/SYSTEM.md, AND extensions — is admissible to run inside the contained cage**, under two invariants:

- **(a) per-asset `ro`/`rw`, `ro` DEFAULT, host-owned (evolved 2026-06-17 — wlwc D4).** Each substrate asset (per skill subfolder, per extension, per hook) carries a **mount mode**. `ro` (the published default) = locked: the mount is read-only (no cage→host write-back; a compromised cage can't rewrite host substrate) and the source is host-owned / not-agent-writable (closes the ADR-005 D7 pre-stage threat). `rw` (opt-in, power-user, host-configured) = a **live write-through bind-mount** so the agent improves its own skills/extensions in-cage — edits land on the host, versioned via the host's normal git, no separate in-cage git step. This is the property that already justifies mounting `CLAUDE.md`/CC-skills read-only — not "inertness" (`SYSTEM.md`/`AGENTS.md` auto-inject; skills run scripts). The **same `ro`/`rw` knob is what implements the floor-lock** (D3 / wlwc D5): the guard wiring is simply an *unwritable* asset on its own load path, *separate from* any `rw` self-improvement substrate, so an `rw` skill and an `ro` guard coexist in one cage.

  - **Per-config security posture (wlwc D7).** Uncrossable-from-inside holds **in the `ro`-default config**: no agent-added/edited substrate exists, so command-gating is genuinely uncrossable from inside for bundled agents (CC: managed-settings hook un-suppressible + deny-wins; pi: guard unwritable + no competing extension addable, so the post-approval-mutation "vector b" stays closed). In **opt-in `rw` mode** the guard wiring is *still* unwritable (separate path) so the guard can't be removed — but pi's post-approval-mutation vector REOPENS for an agent-*edited* `rw` extension; this is a **knowing relaxation backstopped by containment** (the operator opted in on their own machine; command-gating is an accident-guardrail above containment; containment bounds the blast radius). The **fail-closed projected-extension validator** (ADR-005 D11 / D3) is **retained** for `rw`-*projected* extensions (rejecting, at compose time, host-projected extensions that register post-approval-mutating interceptors or shadow the guard); the residual is the runtime-*edited* `rw` extension a compose-time validator cannot catch — named as accepted, not papered over.

  - **Note:** read-only projection is *strictly safer* than the former status quo, where the baked guard's own directory was agent-writable (rip-cage-olen — **closed 2026-06-17**: the guard file + its `extensions/` dir are now root-owned). That per-agent root-own-dir machinery is itself being retired and re-done onto the D3/wlwc-D5 unwritable-separate-load-path model (a dir cannot be both root-locked and agent-writable, which is exactly why olen's approach conflicts with `rw` extensions).
- **(b) cannot weaken the welded safety floor.** A *projected* extension is admissible only when it cannot disable or bypass the floor. For the interim (D2) this is established **per-extension by vetting** the specific extension projected (e.g. the `subagent` extension is verified tool-only, no `tool_call` interceptor — fact #3) combined with **read-only** projection (the agent cannot mutate the projected copy) and pi's **monotonic block** (fact #4). The *general* mechanism — a fail-closed validator that rejects any projected extension hook that mutates a command post-approval, shadows the guard, or registers a bypassing interceptor — is built by the D3 seam (the MULTIPLEXER hook-bounds checks at `rc:6766+` are the template). This ADR does NOT assume "the floor is already uncrossable from inside": that holds for the config vector (ADR-025) and, as of **rip-cage-olen (closed 2026-06-17)**, for the **pi** agent-writable-extension vector (the baked guard + its `extensions/` dir are root-owned, so the agent cannot replace the guard nor drop a competing post-approval-mutating extension). The **CC** hook-registration vector (rip-cage-r9n4, open) remains. Making the floor uncrossable-from-inside was a sibling precondition, not a premise of D1; for pi's baked-guard-write vector it is now satisfied (the *projected*-extension case stays per-extension-vetted until the D3 validator).

`CONFIG` (settings/keybindings/trust/themes/models), `CREDENTIALS` (`auth.json` — the lone RW mount, ADR-019 D1), and `RUNTIME-STATE` (sessions/git/npm/logs) stay **cage-owned** — for config-integrity, secret, and mutable-state reasons respectively, unrelated to whether they execute.

This generalizes ADR-019 D1: the host→cage surface is no longer "only auth.json" but "auth.json (RW) + read-only host-owned agent-substrate." The previously-hardcoded CC projection is retroactively the canonical instance.

**Inline counter-argument (FIRM rigor, ADR-013 D3):** *ADR-019 D1 narrowed the pi host→cage surface to only `auth.json` to shrink attack surface under ADR-024; admitting skills/extensions re-expands it, extensions are executable code that could subvert the guard (verification even confirmed the guard was self-disablable — olen), and now (2026-06-17) admitting `rw` substrate opens a cage→host write-back path a compromised cage could use to rewrite host methodology that auto-loads in the next host session.* **Rebuttal:** ADR-024's threat is the agent following hostile instructions from **attacker-controllable channels it fetches** (web/READMEs/MCP/workspace) — not operator-owned, host-controlled methodology substrate. "Executable code in the cage" is the cage's premise, not a new risk. The guard-subversion concern is handled without assuming an uncrossable floor: in the `ro`-default config the projected extension is read-only (the agent can't tamper with it), it is vetted tool-only / validator-bounded (registers no interceptor), and pi's block is monotonic (no allow-past). On the new `rw` surface: it is **opt-in, host-configured, per-asset** — the `ro` published default has *no* write-back path; an operator who enables `rw` on their own machine knowingly trades a backstop for the self-improvement flywheel, the guard wiring stays unwritable regardless of mode (so `rw` never reaches the guard), and the cage→host edit is bounded by containment + the operator's own opt-in (wlwc D4/D7). A future isolated-copy + reviewed-promote flavor (sourcing from the host's git, not a live bind-mount) is the named mitigation if the live flavor proves too sharp — tracked, not built now. The agent-writable-guard weakness (olen) is *pre-existing and independent* of this projection; olen's fix (**done for pi 2026-06-17**, being re-done onto the D3/wlwc-D5 unwritable-separate-path model) closes the baked-guard-write vector, and the D3 validator closes the *projected*-extension case generally. The real anomaly the narrow D1 reading created is the CC/pi asymmetry.

**What would invalidate D1:** if a *read-only, vetted, tool-only* projected extension could still weaken the floor (e.g. pi's block turns out non-monotonic, or extension load can preempt the guard despite vetting), invariant (b)'s interim basis fails and extensions must be baked-root-owned only, not mountable. OR (the `rw` surface, wlwc D4/D7): if live `rw` write-through proves a real cage→host escalation **even on the operator's own machine** (not merely a theoretical write-back), the live `rw` flavor is replaced by isolated-copy + reviewed-promote; OR if the published default is ever shipped `rw`, or the `ro`-default's vector-b closure does not actually hold (pi loads agent-writable extensions even in `ro` mode), the default-safe posture fails.

| Alternative | Rejected because |
|---|---|
| Keep ADR-019 D1 "only auth.json"; bake everything | `reasoned:` methodology substrate changes constantly (edited this session) → image rebuild per edit; and leaves the CC/pi asymmetry unprincipled. |
| Exclude executable extensions; mount only "inert" content | `direct:` skills already carry executable scripts (CC + pi), and pi can't spawn sub-agents without the `subagent` extension (verified fact #2) — so "inert-only" neither matches reality nor unblocks the factory. |
| All-`ro` substrate, never writable (the former invariant (a)) | `direct:` (reframed 2026-06-17 — wlwc D4) blocks the self-improvement flywheel: an agent improving its own skills/extensions in-cage is a core maintainer workflow and a product feature that read-only-from-host forecloses. `rw` is now **admitted, opt-in, per-asset, `ro`-default** — the all-`ro` rule is too coarse to express "rw skills + ro guard in one cage," which is exactly the floor-lock requirement. |
| Admit substrate read-write *globally / by default* (blanket parity with auth.json) | `reasoned:` a blanket-`rw` default would open an unbounded cage→host write-back path for every cage. The admitted form is **per-asset, host-configured, `ro` by default** — write-through is a knowing operator opt-in on their own machine, backstopped by containment, with the guard wiring unwritable regardless of mode (wlwc D4/D7); a global default-`rw` is what stays rejected, not `rw` itself. |

### D2 (FLEXIBLE — being RETIRED by the now-FIRM D3 seam, 2026-06-17) — pi substrate (instruction-content + the `subagent` extension) projected as a hardcoded per-bundled-agent block, symmetric with CC (interim)

> **Retirement status (2026-06-17, wlwc):** D3 has been lifted EXPLORATORY→FIRM, so the seam that absorbs this interim is now being **built**, not deferred. D2 is **acknowledged interim debt on a now-active retirement path** — the hardcoded CC + pi projection blocks migrate onto the D3 manifest-derived CONTENT-MOUNT seam, and this decision retires when that migration lands (no half-migrated state — pi's olen root-own-dir machinery is re-done onto the unwritable-separate-path model in the same program; wlwc D5/D6). The shipped interim below remains an accurate description of current state until the migration completes.

Project pi's instruction-content (skills, prompts, roles, AGENTS.md, SYSTEM.md/APPEND_SYSTEM.md seam) **and** dotpi's `subagent` extension into the cage, read-only, so `/skill:send-it` and `/drover` both *resolve and run*. The shipped interim (rip-cage-kstk) implements this as a **hardcoded per-bundled-agent projection block in `rc`** — a presence-gated (`[[ -d ~/.pi/agent ]]`) loop that is data-driven over *that agent's asset table* (`{asset → cage-path}` tuples), with **no `if/elif` branch on agent identity** beyond the presence gate. This is **symmetric with the pre-existing hardcoded CC projection block** (the `~/.claude/{skills,commands,agents}` mounts), which is the same shape.

This is **D12-clean by classification, not by manifest-derivation**: ADR-005 D12 forbids `rc` naming a specific *optional* tool (the herdr-leak anti-pattern); `claude` and `pi` are **bundled cage-infra** (per D1, agent-substrate projection is cage-infrastructure), so a hardcoded bundled-agent block does not trip D12. The interim is therefore **NOT yet manifest-derived** — `rc` does not read the seeded manifest to discover which agents declare substrate; that, and a per-agent substrate *declaration* schema, is the **D3 target** (the CONTENT-MOUNT archetype + manifest-derived allowed-set, isomorphic to the multiplexer allowed-set).

Mechanism per asset: resolve host realpath (the dotpi symlinks) → bind-mount `:ro` → `/home/agent/.rc-context/pi-<asset>` → ADR-023 denylist check (warn-and-skip, incidental tier) → symlink into `~/.pi/agent/<asset>` (pi discovers via `PI_CODING_AGENT_DIR`). The `subagent` extension is projected **selectively** (just that extension, vetted tool-only per fact #3) so the baked `dcg-gate` is not shadowed.

**Rationale:** projecting a bundled agent's substrate is cage-infrastructure, not optional-tool composition; the **bundled-vs-optional classification** (not manifest-derivation) is what keeps the hardcoded block D12-clean. This is **acknowledged interim debt** — pre-seam hardcode, NOT "floor", NOT a D12 exception — to be absorbed by D3. (A fresh-context reviewer read the hardcoded `pi` block as a D12 violation precisely because the earlier D2 prose promised manifest-derivation the interim never implemented; this revision removes that trap.)

**What would invalidate D2:** D3's seam ships → CC + pi migrate onto it and this retires.

### D3 (FIRM — lifted EXPLORATORY→FIRM 2026-06-17, wlwc) — a composable agent-substrate seam carrying ALL substrate types incl. extensions, with the floor-lock as a declared per-agent slot

Lift BOTH the hardcoded CC projection and pi onto the composable pattern of `session.multiplexer` (ADR-021 D6) and `network.egress.mediator` (ADR-026 D5): agents become **manifest-declared configurable entries**, and their substrate — skills/prompts/roles **and extensions** — projects via a new **CONTENT-MOUNT archetype** (per-asset `ro`/`rw` mount, `ro` default per D1/wlwc-D4 + symlink-into-config; for extensions, + a **fail-closed floor-protection validator** that rejects projected hooks which mutate commands post-approval, shadow the guard, or register bypassing interceptors). Config selection + manifest-derived allowed-set + baked label + zero `rc` edits to add an agent.

**The floor-lock is a declared per-agent slot (wlwc D5), replacing olen's per-agent root-own-dir machinery.** The guard wiring for any agent is declared as an **agent-unwritable asset on its own load path** (managed-settings for CC / baked-root-owned for pi / `ro`-mounted for host-supplied agents), **separate from the `rw` substrate** — and the dcg-guard wrapper/engine stays root-owned. A fail-loud init check is retained and (per the FIRM repo convention in `.claude/harness.md`, "validate a baked safety artifact's OWNERSHIP, not just its presence") asserts the guard path is root-owned / non-agent-writable, not merely present (presence-only is fail-open against unlink+recreate). The **enforced property** is "guard path unwritable + on a separate path from `rw` substrate", not "the whole extensions dir is root-chowned" — so olen's per-agent ownership machinery (root-own pi's whole `extensions/` dir + sudo dance) is **retired and re-done onto this model**, because a dir cannot be both root-locked and agent-writable, which is exactly what conflicts with `rw` self-improvement.

**Nuance:** agent-substrate is **additive** (all present agents project simultaneously, TOOL-like set semantics), unlike single-select multiplexer/mediator. This is the "enable agent extensions, cleanly" design — extensions are admitted as *validated* providers, not excluded. The validator here is what makes invariant (b) structural rather than per-extension-vetted, and it is **retained for `rw`-projected extensions** (D1/wlwc-D7).

Anchors a follow-on **epic isomorphic to ta1o.5** (build seam → migrate CC → migrate pi off the D2 interim debt → re-do the olen floor-lock onto the declared-per-agent-slot model). Relates to rip-cage-olen (the guard-integrity fix the validated seam assumes — **closed 2026-06-17 for pi**, being re-done onto the unwritable-separate-path model) and rip-cage-r9n4 (the open CC instance, closing early via managed-settings as the first floor-lock instance, wlwc D6).

**Rationale (lift warrant):** rip-cage is **published for heterogeneous harnesses** — the maintainer runs CC + pi, but publishes rip-cage for users on codex, crush, and other agents — so "pluggable agents" is a product requirement, not speculative generality. The rule-of-two objection to building the seam (the EXPLORATORY basis) no longer holds: the third/fourth/fifth agents are the published users' harnesses. And the per-agent floor-lock machinery (olen) does not generalize — it root-owned a whole directory + needed a sudo dance to place even one extension symlink, which does not scale to N projected/editable extensions and directly conflicts with letting agents improve their own extensions in-cage; it was about to be copy-pasted into r9n4 as a second snowflake. The declared-per-agent-slot floor-lock + CONTENT-MOUNT archetype dissolve both.

**Alternatives considered:**

| Alternative | Rejected because |
|---|---|
| Keep CC/pi hardcoded interim, defer the seam (the EXPLORATORY status quo) | `reasoned:` the publish-for-other-harnesses requirement removes the "no third agent" gate — the third agent is the users' harnesses, so the seam is load-bearing now, not speculative. |
| Bake a fixed agent enum into `rc` | `direct:` ADR-005 D12 forbids naming optional tools in source; agents become manifest-declared providers with zero `rc` edits to add one. |
| Keep olen's per-agent root-own-dir + sudo-dance floor-lock | `direct:` does not generalize to N agents/extensions, conflicts with `rw` extensions (a dir cannot be both root-locked and agent-writable), and was about to become a second snowflake at r9n4 — replaced by the declared-per-agent-slot (unwritable separate load path). |

**What would invalidate D3:** if the agent-provider contract cannot express a real target harness's wiring (some agent has no command-interception mechanism at all), that agent gets containment-only (no command-gating) — an accepted gap (containment is the floor; the command-guard is a best-effort accident-guardrail per agent); but if that is the *common* case, the seam's command-guard value is reduced.

## Consequences

- D2 *actually unblocks* the in-cage drover dogfood (dotpi-3bi.8/.4) — skills resolve AND sub-agent dispatch runs — without weakening the floor.
- The CC/pi asymmetry is named interim debt with a now-active (D3 FIRM, 2026-06-17) retirement path: build the CONTENT-MOUNT seam → migrate CC → migrate pi off the D2 interim → re-do the olen floor-lock onto the declared-per-agent-slot model.
- Spawned child `pi` processes inherit the guard (they load the baked `dcg-gate.ts` via the inherited `PI_CODING_AGENT_DIR`); the agent-writable-guard weakness once tracked in rip-cage-olen is **closed for pi** (root-owned guard + `extensions/` dir).
- The "is executable substrate admissible?" question is resolved (yes, under containment + read-only/host-owned + per-extension vetting now / validator later), dissolving the earlier executable-exclusion.
